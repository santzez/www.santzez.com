1. Distribuciones Linux de servidor

Linux, en sentido estricto, es solo el núcleo (kernel, Tema 9). Lo que se instala y usa en la práctica es una distribución (distro): el núcleo Linux empaquetado junto con un gestor de paquetes, un conjunto de herramientas del sistema y unas configuraciones por defecto. Las distribuciones de servidor más usadas se agrupan en familias, según el gestor de paquetes y el linaje del que derivan.

Familia Distribuciones de servidor habituales Gestor de paquetes
Debian / Ubuntu Debian, Ubuntu Server (LTS) APT (paquetes .deb)
Red Hat RHEL, Rocky Linux, AlmaLinux, Fedora DNF (paquetes .rpm)
SUSE SUSE Linux Enterprise Server, openSUSE Zypper (paquetes .rpm)
🔍 Para ampliar — contexto útil, menor probabilidad de pregunta directa
Red Hat descontinuó CentOS Linux como distribución estable gratuita de larga duración (CentOS Linux 7 llegó a su fin de vida en junio de 2024), reorientando el proyecto hacia CentOS Stream, una rama de desarrollo previa a RHEL, no equivalente en estabilidad a la antigua CentOS. En su lugar, la comunidad impulsó dos alternativas gratuitas compatibles con RHEL: Rocky Linux y AlmaLinux, hoy las opciones de referencia cuando se busca una distribución de la familia Red Hat sin coste de licencia. Ubuntu Server, con sus versiones LTS (Long Term Support, publicadas cada dos años y con varios años de soporte garantizado), es probablemente la opción más extendida como servidor de propósito general.

2. Instalación de un servidor Linux

La instalación de un servidor Linux sigue la misma filosofía minimalista que Windows Server Core (Tema 12): se instala solo lo estrictamente necesario, sin entorno gráfico de escritorio, para reducir tanto la superficie de ataque como el consumo de recursos y la cantidad de actualizaciones a aplicar. La administración se realiza en línea de comandos, habitualmente a través de una shell Bash.

  • Instalación manual: a partir de una imagen ISO de instalación, respondiendo de forma interactiva a las preguntas del instalador (particionado, red, usuario inicial…).
  • Instalación desatendida: automatiza estas respuestas mediante un fichero de configuración (kickstart en la familia Red Hat; preseed o, cada vez más, cloud-init en Debian/Ubuntu), lo que resulta imprescindible para desplegar muchos servidores de forma idéntica y reproducible, especialmente en entornos virtualizados (Tema 10) o en la nube.

Particionado y sistemas de archivo

Junto al particionado tradicional (dividir el disco en particiones de tamaño fijo), es habitual usar LVM (Logical Volume Manager): una capa de abstracción que agrupa uno o varios discos físicos en un «volumen» del que se pueden crear particiones lógicas redimensionables en caliente, con mucha más flexibilidad que el particionado clásico. Los sistemas de archivo más habituales en servidores Linux son ext4 (el más extendido por defecto), XFS (frecuente en la familia Red Hat, con buen rendimiento en archivos de gran tamaño) y Btrfs (con funciones avanzadas como instantáneas o subvolúmenes); el concepto general de sistema de archivos como gestor de la información persistente ya se introdujo en el Tema 9.

3. Gestión de paquetes y del software

El software en Linux se instala, actualiza y elimina mediante gestores de paquetes, que descargan el software desde repositorios (servidores remotos con las versiones disponibles de cada paquete) y resuelven automáticamente sus dependencias (otros paquetes necesarios para que el software funcione), a diferencia de instalar un paquete suelto de bajo nivel (`.deb` con `dpkg`, `.rpm` con `rpm`) directamente, donde esas dependencias hay que resolverlas a mano.

Acción Debian / Ubuntu (APT) Red Hat / Rocky / AlmaLinux (DNF)
Actualizar el índice de paquetes disponibles apt update dnf check-update
Actualizar los paquetes instalados apt upgrade dnf upgrade
Instalar un paquete apt install <paquete> dnf install <paquete>
Formato de paquete .deb .rpm
⚠️ No confundir
No debe confundirse «actualizar el índice» con «actualizar los paquetes». El primer paso (apt update / dnf check-update) solo refresca la lista de versiones disponibles en los repositorios configurados; no instala nada. El segundo paso (apt upgrade / dnf upgrade) es el que realmente descarga e instala las versiones más recientes de los paquetes ya presentes en el sistema. Olvidar el primer paso es un error habitual que lleva a instalar versiones desactualizadas sin darse cuenta.

4. Arranque y gestión de servicios: systemd

systemd es el sistema de inicio (init system) y la infraestructura de gestión de servicios adoptada como estándar por la gran mayoría de distribuciones Linux actuales (Ubuntu, Debian, RHEL y derivados, SUSE), en sustitución de los antiguos SysV init y Upstart. Es el primer proceso que arranca tras el núcleo (identificado siempre con el PID 1) y se encarga de iniciar, detener y supervisar el resto de servicios del sistema.

  • Unidades (units): ficheros de configuración que describen un recurso gestionado por systemd; las más comunes son las unidades `.service` (un servicio o demonio), aunque también existen `.socket`, `.mount` o `.timer` (para tareas programadas, alternativa moderna a cron).
  • systemctl: comando principal para gestionar servicios: `systemctl start|stop|restart|status <servicio>` para controlar su ejecución en el momento, y `systemctl enable|disable <servicio>` para que arranque (o no) automáticamente con el sistema.
  • journalctl: herramienta para consultar el registro centralizado de sucesos (el «journal») que recopila systemd de todos los servicios y del propio arranque del sistema.
⚠️ No confundir
systemd no es el núcleo de Linux, sino el primer proceso de espacio de usuario que el núcleo arranca. Cumple, dentro de un servidor Linux, un papel equivalente al que cumplen conjuntamente el Administrador de servicios (services.msc) y, en parte, el programador de tareas de Windows (Tema 11), pero es importante no llamarlo «núcleo» ni confundirlo con él en el examen.

5. Administración básica y acceso remoto

La administración de un servidor Linux es, salvo excepciones, remota y en línea de comandos, a través de SSH (Secure Shell): un protocolo que cifra toda la comunicación entre el cliente y el servidor, y que escucha por defecto en el puerto 22. La autenticación puede hacerse por contraseña o, de forma más segura y habitual en producción, mediante un par de claves criptográficas (pública en el servidor, privada en el cliente).

Todo sistema Linux distingue entre el superusuario root, con privilegios totales sobre el sistema, y los usuarios normales, con privilegios limitados. La buena práctica de seguridad no es iniciar sesión directamente como root, sino hacerlo como un usuario normal y usar el comando sudo ("superuser do") para ejecutar puntualmente, y de forma auditable, comandos concretos con privilegios elevados.

6. Mantenimiento básico: actualizaciones

Además de la actualización manual con el gestor de paquetes (punto 3), la mayoría de distribuciones ofrecen mecanismos de actualización automática de seguridad: `unattended-upgrades` en Debian/Ubuntu o `dnf-automatic` en la familia Red Hat, que instalan sin intervención manual, al menos, los parches de seguridad críticos.

Como en cualquier sistema operativo, cada distribución tiene un ciclo de vida y una fecha de fin de soporte: las versiones LTS de Ubuntu, por ejemplo, reciben cinco años de soporte estándar y, opcionalmente, cinco años adicionales de mantenimiento de seguridad extendido (ESM). El mismo principio que se señaló para Windows 10 en el Tema 11 aplica aquí: seguir usando una versión sin soporte deja el servidor sin parches de seguridad, por lo que la migración a una versión soportada debe planificarse con antelación.

🔍 Para ampliar — contexto útil, menor probabilidad de pregunta directa
Actualizar el núcleo Linux normalmente exige reiniciar el servidor para que la nueva versión entre en funcionamiento. Algunas distribuciones de la familia Red Hat ofrecen mecanismos de «live patching» (por ejemplo, kpatch), que permiten aplicar parches de seguridad críticos al núcleo en caliente, sin reiniciar, retrasando el reinicio a una ventana de mantenimiento posterior. Es una técnica avanzada, orientada a servicios que no pueden permitirse una parada no planificada.

7. Monitorización

Un servidor Linux se monitoriza principalmente a través de comandos y utilidades en línea de comandos, complementados con el registro centralizado de systemd (journalctl) y los archivos de registro clásicos en `/var/log`.

Aspecto a monitorizar Comando / herramienta Qué muestra
Procesos y consumo de CPU/memoria en tiempo real top / htop Listado interactivo de procesos y su consumo de recursos
Listado de procesos ps Instantánea de los procesos en ejecución
Memoria disponible free Memoria RAM y swap usadas y libres
Espacio en disco df Espacio ocupado y libre por sistema de archivos
Uso de espacio por directorio du Tamaño ocupado por carpetas y archivos
Carga del sistema y tiempo activo uptime Tiempo desde el arranque y la «load average»
Conexiones y puertos de red ss (o el más antiguo netstat) Conexiones de red activas y puertos en escucha
Registro de sucesos del sistema journalctl / /var/log Histórico de eventos del sistema y de los servicios

Para entornos con muchos servidores, estas herramientas puntuales se complementan con soluciones de monitorización centralizada que recogen métricas de forma continua y permiten configurar alertas, como Nagios, Zabbix o la combinación Prometheus + Grafana; su configuración detallada queda fuera del alcance de este tema introductorio.

📌 Nivel básico — imprescindible para el examen
Resulta útil comparar las herramientas de Linux con sus equivalentes de Windows Server, vistas en el Tema 12: journalctl / /var/log equivalen al Visor de eventos; top / htop equivalen al Administrador de tareas; systemctl equivale, a grandes rasgos, a la consola de servicios (services.msc). Los conceptos que hay detrás son los mismos en ambos sistemas operativos; lo que cambia es la herramienta concreta y su sintaxis.

8. Resumen esquemático

  • Distribuciones: familias Debian/Ubuntu (APT, .deb), Red Hat (DNF, .rpm — Rocky Linux y AlmaLinux como sucesores gratuitos de CentOS) y SUSE (Zypper); Ubuntu Server LTS como opción muy extendida de propósito general.
  • Instalación: filosofía minimalista, sin entorno gráfico; instalación manual o desatendida (kickstart / preseed / cloud-init); LVM para particionado flexible; sistemas de archivo ext4, XFS y Btrfs.
  • Gestión de paquetes: repositorios y resolución automática de dependencias; «actualizar el índice» (apt update) no es lo mismo que «actualizar los paquetes» (apt upgrade).
  • systemd: sistema de inicio y gestor de servicios estándar (PID 1); unidades .service gestionadas con systemctl; registro consultable con journalctl.
  • Administración: acceso remoto por SSH (cifrado, puerto 22, preferible autenticación por clave); usuario root frente a usuarios normales con sudo.
  • Actualizaciones: actualización manual o automática (unattended-upgrades / dnf-automatic); ciclo de vida y fin de soporte de cada distribución, igual de relevante que en Windows; live patching del núcleo en casos avanzados.
  • Monitorización: top/htop, ps, free, df, du, uptime, ss y journalctl/var/log como herramientas básicas; Nagios, Zabbix o Prometheus+Grafana para monitorización centralizada a mayor escala.
