1. Concepto de virtualización

La virtualización es la tecnología que permite crear una versión virtual —simulada por software— de un recurso físico, de forma que ese recurso pueda compartirse entre varios usos independientes entre sí como si cada uno dispusiera de su propio recurso dedicado. Aplicada a equipos y servidores, la virtualización permite ejecutar varios ordenadores «completos» de forma simultánea sobre un único equipo físico, cada uno con su propio sistema operativo, funcionando de forma aislada de los demás.

En este contexto se manejan tres términos clave:

  • Host (anfitrión): el equipo físico real, con su hardware (CPU, memoria, disco, red), sobre el que se ejecuta la virtualización.
  • Guest (invitado): cada sistema operativo que se ejecuta de forma virtualizada sobre el host.
  • Máquina virtual (VM): el «ordenador» definido por software —con su CPU virtual, memoria virtual, disco virtual y tarjeta de red virtual— sobre el que se instala y ejecuta cada sistema operativo invitado.
📌 Nivel básico — imprescindible para el examen
Es como dividir un edificio (el host) en varios apartamentos independientes (las máquinas virtuales): cada inquilino (sistema operativo invitado) tiene sus propias habitaciones y su propia puerta con llave, y no ve lo que ocurre en los demás apartamentos, aunque todos compartan la misma estructura, las mismas tuberías y el mismo suministro eléctrico del edificio (el hardware físico del host).

2. El hipervisor: concepto y tipos

El hipervisor (también llamado VMM, Virtual Machine Monitor) es la capa de software responsable de crear, ejecutar y gestionar las máquinas virtuales, repartiendo entre ellas de forma controlada los recursos físicos del host y garantizando su aislamiento.

  • Hipervisor de tipo 1 (bare-metal o «nativo»): se instala directamente sobre el hardware, sin un sistema operativo convencional de por medio; el propio hipervisor actúa como capa base del sistema. Ofrece mejor rendimiento y es el modelo habitual en entornos de servidor y producción. Ejemplos: VMware ESXi, Microsoft Hyper-V, KVM (integrado en el núcleo de Linux), Xen, Proxmox VE.
  • Hipervisor de tipo 2 (alojado o «hosted»): se instala como una aplicación más sobre un sistema operativo convencional (Windows, Linux, macOS) ya existente. Es más sencillo de desplegar, pero introduce una capa adicional que penaliza algo el rendimiento. Habitual en entornos de escritorio, pruebas o desarrollo. Ejemplos: VMware Workstation/Fusion, Oracle VirtualBox, Parallels Desktop.
Criterio Tipo 1 (bare-metal) Tipo 2 (alojado)
Dónde se instala Directamente sobre el hardware Sobre un sistema operativo anfitrión
Rendimiento Mayor: acceso más directo al hardware Menor: capa adicional del SO anfitrión
Uso típico Servidores y producción Escritorio, pruebas y desarrollo
Ejemplos VMware ESXi, Hyper-V, KVM, Xen, Proxmox VE VMware Workstation/Fusion, VirtualBox, Parallels
⚠️ No confundir
Microsoft Hyper-V suele generar dudas al clasificarlo: se activa como un rol dentro de una instalación de Windows Server o Windows 11, lo que podría sugerir que es de tipo 2. Sin embargo, técnicamente es de tipo 1: al habilitarlo, el propio Windows pasa a ejecutarse como una máquina virtual más (la «partición raíz») por encima del hipervisor, que se sitúa directamente sobre el hardware.

3. Tipos de virtualización

  • Virtualización completa (full virtualization): el hipervisor simula un hardware completo, de forma que el sistema operativo invitado se ejecuta sin modificaciones, sin ser consciente de que está virtualizado. Se apoya en extensiones de virtualización por hardware del propio procesador (Intel VT-x, AMD-V), que aceleran esta simulación.
  • Paravirtualización: el sistema operativo invitado sí es consciente de que se ejecuta sobre un hipervisor, y se modifica (o incluye drivers específicos) para comunicarse directamente con él mediante llamadas optimizadas (hypercalls), evitando parte de la simulación de hardware. Puede ofrecer mejor rendimiento, a cambio de requerir un SO invitado compatible.
  • Virtualización a nivel de sistema operativo (contenedores): en lugar de virtualizar hardware completo para ejecutar SO invitados independientes, se aísla a nivel de procesos sobre un único núcleo (kernel) compartido con el host. Cada contenedor empaqueta una aplicación con sus dependencias, pero no un sistema operativo completo, lo que lo hace mucho más ligero y rápido de arrancar que una máquina virtual (Docker es el ejemplo más extendido).
Criterio Máquina virtual (VM) Contenedor
Qué se virtualiza El hardware completo El sistema operativo (procesos aislados)
Sistema operativo Cada VM tiene su propio SO invitado Todos los contenedores comparten el núcleo del host
Peso y arranque Mayor tamaño; arranque en minutos Muy ligero; arranque en segundos
Aislamiento Muy alto (SO completo independiente) Menor que una VM, aunque suficiente para la mayoría de usos

Los contenedores no forman parte del contenido explícito de este tema, centrado en la virtualización de equipos y servidores mediante hipervisores, pero conviene reconocer la diferencia porque es una fuente habitual de confusión: una máquina virtual virtualiza hardware para ejecutar un SO completo; un contenedor aísla procesos sobre un SO ya en ejecución, sin virtualizar hardware.

4. Casos de uso

  • Consolidación de servidores: sustituir varios servidores físicos infrautilizados por un único host más potente que ejecuta varias máquinas virtuales, aprovechando mucho mejor la capacidad de cada equipo.
  • Entornos de prueba y desarrollo: crear y destruir máquinas virtuales bajo demanda para probar software, actualizaciones o configuraciones, sin arriesgar los sistemas de producción y sin necesidad de hardware dedicado para cada prueba.
  • Alta disponibilidad: mediante la migración en caliente de máquinas virtuales entre distintos hosts físicos (por ejemplo, vMotion en VMware o Live Migration en Hyper-V), es posible mover una VM en ejecución de un servidor a otro sin interrupción del servicio, lo que facilita el mantenimiento planificado del hardware.
  • Recuperación ante desastres (DR): replicar máquinas virtuales completas hacia otro centro de datos permite recuperar el servicio con rapidez ante un fallo grave o una catástrofe en el centro principal.
  • Escritorios virtuales (VDI, Virtual Desktop Infrastructure): alojar los escritorios de los usuarios como máquinas virtuales centralizadas en el centro de datos, a las que se accede de forma remota, facilitando su gestión y actualización centralizada.
  • Base de la computación en la nube: los proveedores de nube pública construyen sus servicios sobre virtualización a gran escala, permitiendo ofrecer «servidores virtuales» bajo demanda (el detalle de la nube se estudia en el Tema 32).

5. Ventajas y desventajas

Ventajas

  • Mejor aprovechamiento del hardware: varios sistemas comparten los mismos recursos físicos en lugar de infrautilizar servidores dedicados.
  • Ahorro de costes: menos servidores físicos que comprar, alojar, alimentar y refrigerar en el centro de datos (CPD).
  • Aislamiento: en condiciones normales, un fallo o una incidencia de seguridad dentro de una máquina virtual no debería afectar a las demás VM del mismo host.
  • Flexibilidad y rapidez de despliegue: crear, clonar o eliminar una máquina virtual es cuestión de minutos, frente a los plazos de adquisición e instalación de un servidor físico nuevo.
  • Instantáneas (snapshots): permiten capturar el estado completo de una VM en un momento dado y volver a él rápidamente, muy útil antes de aplicar cambios arriesgados.
  • Migración en caliente: mover una VM en ejecución entre hosts físicos sin cortar el servicio, facilitando el mantenimiento sin tiempos de inactividad.

Desventajas

  • Sobrecarga de rendimiento (overhead): la capa de virtualización consume recursos y puede introducir una ligera pérdida de rendimiento frente al hardware físico, aunque las extensiones de virtualización por hardware (VT-x, AMD-V) la han reducido notablemente.
  • Punto único de fallo: si el host físico falla y no existe una configuración de alta disponibilidad, todas las máquinas virtuales que aloja se ven afectadas a la vez, a diferencia de un fallo en un servidor físico dedicado, que solo afecta a ese servicio.
  • Mayor complejidad de gestión: requiere planificar con cuidado el reparto de recursos entre VM; un sobreaprovisionamiento (overcommit) mal controlado de CPU o memoria puede degradar el rendimiento de todas las máquinas virtuales del host.
  • Coste de licenciamiento: el software de virtualización de nivel empresarial suele requerir licencias, cuyo coste ha cobrado especial relevancia en los últimos años (véase el punto 6).

6. Panorama actual de la virtualización

Durante años, VMware (con su hipervisor vSphere/ESXi) fue el proveedor de referencia en virtualización de servidores empresariales. Tras su adquisición por Broadcom, completada en 2023, el modelo de licenciamiento cambió de forma significativa: desde enero de 2024 dejaron de venderse licencias perpetuas para cualquier producto VMware, sustituidas por un modelo exclusivamente de suscripción, con precios calculados por núcleo de CPU (frente al anterior modelo por zócalo/socket) y unos mínimos de compra que han encarecido notablemente el coste para organizaciones con clústeres pequeños o medianos.

Este cambio ha impulsado a muchas organizaciones a evaluar alternativas: Microsoft Hyper-V (integrado en Windows Server, sin coste de licencia adicional para quien ya dispone de esas licencias), y opciones de código abierto basadas en KVM (Kernel-based Virtual Machine, integrado en el núcleo de Linux), habitualmente gestionadas a través de plataformas como Proxmox VE.

🔍 Para ampliar — contexto útil, menor probabilidad de pregunta directa
Este panorama comercial cambia con rapidez y no conviene memorizar cifras concretas de precios, sino entender la tendencia de fondo: el mercado de la virtualización, tradicionalmente dominado por una única solución de referencia, se ha vuelto más fragmentado, con alternativas de pago (Hyper-V) y de código abierto (KVM/Proxmox VE) ganando peso frente al modelo clásico de VMware. Los conceptos fundamentales —hipervisor, máquina virtual, snapshot, migración en caliente— son comunes a todas las soluciones, sea cual sea el fabricante.

7. Resumen esquemático

  • Virtualización: crear versiones virtuales de un recurso físico; host (físico), guest (SO virtualizado), VM (el «ordenador» definido por software).
  • Hipervisor: capa que crea y gestiona las VM. Tipo 1 (bare-metal, sobre el hardware, mayor rendimiento: ESXi, Hyper-V, KVM) frente a tipo 2 (alojado sobre un SO, más sencillo: VirtualBox, VMware Workstation).
  • Tipos de virtualización: completa (SO invitado sin modificar, apoyada en VT-x/AMD-V), paravirtualización (SO invitado adaptado, más eficiente) y contenedores (aislamiento de procesos sobre un kernel compartido, sin virtualizar hardware).
  • Casos de uso: consolidación de servidores, pruebas y desarrollo, alta disponibilidad (migración en caliente), recuperación ante desastres, escritorios virtuales (VDI) y base de la computación en la nube.
  • Ventajas: mejor aprovechamiento del hardware, ahorro de costes, aislamiento, rapidez de despliegue, snapshots y migración en caliente.
  • Desventajas: overhead de rendimiento, punto único de fallo del host, mayor complejidad de gestión y coste de licenciamiento.
  • Panorama actual: fin de las licencias perpetuas de VMware (Broadcom, desde 2024) y auge de alternativas como Hyper-V y KVM/Proxmox VE.
