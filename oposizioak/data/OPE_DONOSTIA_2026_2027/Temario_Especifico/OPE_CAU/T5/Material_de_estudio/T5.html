1. De los ficheros al Sistema Gestor de Bases de Datos (SGBD)

En el Tema 4 vimos los ficheros como forma de almacenar información de manera persistente. Cuando varias aplicaciones necesitan compartir esos datos, gestionar ficheros por separado empieza a dar problemas: la misma información se acaba duplicando en varios sitios (redundancia), esas copias se desincronizan entre sí (inconsistencia), y resulta difícil controlar quién accede a qué y evitar que dos procesos escriban a la vez sobre el mismo dato.

Para resolver esto surge el Sistema Gestor de Bases de Datos (SGBD o DBMS, Database Management System): el software que se sitúa entre las aplicaciones y los datos, centralizando su almacenamiento y ofreciendo mecanismos de control que un conjunto de ficheros sueltos no puede garantizar por sí mismo.

  • Base de datos: conjunto organizado de datos, estructurados y relacionados entre sí, almacenados de forma centralizada y gestionados por un SGBD.
  • SGBD: el software que crea, mantiene y controla el acceso a la base de datos, actuando de intermediario entre las aplicaciones y los datos.

Ventajas de una base de datos frente a ficheros independientes

  • Menor redundancia: los datos se almacenan una sola vez y se comparten entre aplicaciones.
  • Mayor integridad y consistencia: el SGBD aplica reglas que evitan datos contradictorios.
  • Control centralizado y seguridad: se puede definir con detalle quién accede a qué información y con qué permisos.
  • Acceso concurrente controlado: varios usuarios o procesos pueden trabajar a la vez sin pisarse los cambios.
  • Independencia entre datos y programas: las aplicaciones no necesitan conocer cómo se almacenan físicamente los datos para poder usarlos.

2. Tipos de bases de datos

No todas las bases de datos organizan la información del mismo modo. Los modelos más habituales son:

  • Jerárquica: organiza los datos en forma de árbol, con relaciones de padre a hijo (una tabla puede tener varios hijos, pero cada hijo tiene un único padre).
  • En red: similar a la jerárquica, pero permite que un registro tenga varios «padres», representando relaciones más complejas mediante punteros.
  • Relacional: organiza los datos en tablas vinculadas mediante claves. Es, con diferencia, el modelo más extendido hoy en día, y el que se desarrolla en el resto de este tema.
  • Orientada a objetos: almacena los datos como objetos, con sus atributos y comportamientos, de forma coherente con la programación orientada a objetos.
  • NoSQL («Not Only SQL»): familia de bases de datos no relacionales, pensadas para grandes volúmenes de datos, alta disponibilidad y esquemas más flexibles que el rígido esquema de tablas del modelo relacional.
📌 Nivel básico — imprescindible para el examen
Para el examen, lo importante es identificar el modelo dominante: la base de datos relacional organiza la información en tablas con filas y columnas, relacionadas mediante claves.
El resto de modelos (jerárquico, en red, orientado a objetos, NoSQL) conviene reconocerlos por su idea general y diferenciarlos del relacional, sin necesidad de memorizar más detalle.
🔍 Para ampliar — contexto útil, menor probabilidad de pregunta directa
Dentro de NoSQL existen varias familias según cómo almacenan la información: clave-valor (por ejemplo, Redis), documental (por ejemplo, MongoDB), columnar (por ejemplo, Cassandra) y de grafos (por ejemplo, Neo4j).
No hace falta memorizar estos ejemplos concretos: basta con saber que NoSQL no es un único modelo, sino una familia de alternativas al modelo relacional.

3. El modelo relacional: conceptos básicos

El modelo relacional fue propuesto por Edgar F. Codd en 1970 y se basa en representar toda la información mediante tablas (también llamadas relaciones), cada una de ellas dedicada a un tipo de entidad (por ejemplo, «empleados» o «departamentos»).

  • Tabla (o relación): estructura bidimensional que almacena los datos de una entidad, organizados en filas y columnas.
  • Fila (o tupla, o registro): cada una de las ocurrencias concretas almacenadas en la tabla.
  • Columna (o atributo, o campo): cada una de las características que describen a la entidad.
  • Dominio: el conjunto de valores posibles que puede tomar un atributo (por ejemplo, el dominio del atributo «edad» son los números enteros positivos razonables para una persona).
📌 Nivel básico — imprescindible para el examen
El modelo relacional reutiliza, con otro nombre, el mismo vocabulario del Tema 4: tabla = fichero, fila = registro, columna = campo.
Pensarlo así ayuda a no tratar este tema como algo completamente nuevo, sino como una forma más estructurada y potente de organizar la misma idea de fondo.

4. Claves y relaciones entre tablas

  • Clave primaria (PK, Primary Key): campo, o conjunto de campos, que identifica de forma única cada fila de una tabla. No puede repetirse ni quedar vacía (nula).
  • Clave foránea (o ajena, FK, Foreign Key): campo de una tabla que referencia la clave primaria de otra tabla (o de la misma), estableciendo así una relación entre ambas.
  • Integridad referencial: regla que garantiza que el valor de una clave foránea corresponde siempre a un valor existente en la clave primaria a la que hace referencia (o que queda vacío, si eso está permitido).

Tipos de relación entre tablas

  • Uno a uno (1:1): cada fila de una tabla se relaciona con, como mucho, una fila de la otra.
  • Uno a muchos (1:N): una fila de una tabla puede relacionarse con varias filas de otra, pero cada fila de la segunda solo se relaciona con una de la primera (por ejemplo, un departamento tiene varios empleados, pero cada empleado pertenece a un único departamento).
  • Muchos a muchos (N:M): varias filas de una tabla pueden relacionarse con varias filas de otra. Se resuelve mediante una tabla intermedia que combina las claves primarias de ambas tablas.
⚠️ No confundir
El diseño detallado de bases de datos (normalización, formas normales, sintaxis SQL) no forma parte de un tema específico de este temario.
Quédate con la idea de fondo: cómo se identifica de forma única una fila (clave primaria) y cómo se conectan las tablas entre sí (clave foránea y tipos de relación), sin entrar a diseñar ni consultar bases de datos reales.

5. Estándares de conectividad

Una aplicación no siempre está escrita en el mismo entorno que el motor de base de datos al que necesita acceder. Para no depender de un SGBD concreto, existen estándares de conectividad: interfaces comunes que permiten a una aplicación dialogar con distintos motores de base de datos de manera uniforme, delegando en un driver específico las particularidades de cada uno.

  • ODBC (Open Database Connectivity): estándar de conectividad de propósito general, que permite a cualquier aplicación acceder a distintos SGBD a través de un driver ODBC específico para cada motor, usando siempre la misma API común.
  • JDBC (Java Database Connectivity): el equivalente a ODBC pensado específicamente para aplicaciones Java, mediante drivers JDBC que permiten conectar con distintos motores de base de datos desde código Java.
🔍 Para ampliar — contexto útil, menor probabilidad de pregunta directa
Tanto ODBC como JDBC comparten la misma idea de arquitectura: la aplicación habla con una API común, y es el driver correspondiente el que traduce esas peticiones al lenguaje propio de cada SGBD.
Hoy en día muchas aplicaciones añaden una capa adicional (ORM, mapeo objeto-relacional) por encima de estos estándares, pero no hace falta profundizar en ello para este tema.

6. Resumen esquemático

  • Base de datos = conjunto organizado de datos centralizado y compartido, gestionado por un SGBD, que reduce redundancia y mejora integridad, seguridad y concurrencia frente a ficheros independientes.
  • Tipos de bases de datos: jerárquica, en red, relacional (la dominante), orientada a objetos y NoSQL (clave-valor, documental, columnar, grafos).
  • Modelo relacional (Codd, 1970): tabla/relación = fichero, fila/tupla = registro, columna/atributo = campo, dominio = valores posibles de un atributo.
  • Clave primaria = identifica de forma única cada fila. Clave foránea = referencia la clave primaria de otra tabla y crea la relación.
  • Tipos de relación entre tablas: uno a uno (1:1), uno a muchos (1:N) y muchos a muchos (N:M, resuelta con tabla intermedia).
  • Estándares de conectividad: ODBC (propósito general) y JDBC (específico de Java), ambos basados en una API común más un driver específico de cada SGBD.
