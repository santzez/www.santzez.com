1. Concepto y funciones de un sistema operativo

Un sistema operativo (SO) es el software de sistema que actúa de intermediario entre el hardware del ordenador y el usuario o las aplicaciones que se ejecutan sobre él. Se carga al arrancar el equipo y permanece en ejecución mientras este funciona, gestionando todos sus recursos: la CPU, la memoria, los dispositivos periféricos (Tema 8) y el almacenamiento.

Clásicamente se explica el papel de un SO desde dos puntos de vista complementarios:

  • SO como gestor de recursos: reparte de forma ordenada y eficiente los recursos limitados del sistema (tiempo de CPU, espacio en memoria, acceso a dispositivos) entre los distintos programas que compiten por ellos, evitando conflictos entre procesos.
  • SO como máquina virtual extendida (o «máquina abstracta»): oculta al programador y al usuario la complejidad real del hardware, ofreciendo una interfaz más sencilla y uniforme (por ejemplo, «guardar un fichero» en lugar de tener que manejar directamente el controlador del disco).

De estas dos funciones generales se derivan las funciones concretas que cualquier sistema operativo cumple: gestión de procesos, gestión de memoria, gestión de la entrada/salida, gestión del sistema de archivos, gestión de la seguridad y de los usuarios, y provisión de una interfaz de usuario (de línea de comandos o gráfica) para interactuar con el sistema.

📌 Nivel básico — imprescindible para el examen
Piensa en el sistema operativo como en el personal de un hotel: los huéspedes (los programas) piden recursos (una habitación, el ascensor, el comedor) y el personal (el SO) organiza el acceso a esos recursos compartidos para que todos puedan usarlos sin chocar entre sí, sin que el huésped necesite saber cómo funciona la caldera o el sistema eléctrico del edificio.

2. Clasificación de los sistemas operativos

No existe un único criterio para clasificar los sistemas operativos: se combinan varios ejes, y un mismo SO puede describirse con varias de estas etiquetas a la vez (por ejemplo, Linux de servidor es multiusuario, multitarea y de tiempo compartido).

Criterio Tipos Ejemplo / comentario
Nº de usuarios simultáneos Monousuario / Multiusuario MS-DOS (monousuario); Linux, Windows Server (multiusuario)
Nº de tareas simultáneas Monotarea / Multitarea MS-DOS (monotarea); Windows, Linux, macOS actuales (multitarea)
Nº de procesadores gestionados Uniprocesador / Multiprocesador SMP (multiprocesador simétrico); sistemas distribuidos
Gestión del tiempo Por lotes / Tiempo compartido / Tiempo real Batch (histórico); time-sharing (uso interactivo actual); RTOS (control industrial)
Ámbito de uso Escritorio / Servidor / Embebido / Móvil Windows 11 (escritorio); Windows Server, Linux (servidor); Android, iOS (móvil)

Dentro de los sistemas de tiempo real (RTOS, Real-Time Operating System) se distingue, además, entre tiempo real duro, donde no cumplir un plazo se considera un fallo del sistema (por ejemplo, el control de un airbag), y tiempo real blando, donde incumplir el plazo degrada el servicio pero no lo invalida (por ejemplo, la reproducción de vídeo en streaming).

⚠️ No confundir
No debe confundirse «tiempo compartido» (time-sharing, el modelo habitual de cualquier SO de escritorio o servidor actual, que reparte la CPU entre varios usuarios o procesos dando sensación de simultaneidad) con «tiempo real» (donde lo importante no es repartir el tiempo, sino garantizar que una tarea concreta se ejecuta antes de un plazo determinado).

3. Arquitectura de un sistema operativo

El núcleo o kernel es la parte central del sistema operativo: el componente que se ejecuta con el máximo nivel de privilegio y que tiene acceso directo al hardware. El resto del software (aplicaciones de usuario, e incluso parte de las utilidades del propio SO) se ejecuta con privilegios más restringidos.

Modos de ejecución

  • Modo núcleo (kernel mode o modo supervisor): acceso completo al hardware y a todas las instrucciones del procesador. Es el modo en el que se ejecuta el núcleo del SO.
  • Modo usuario (user mode): acceso restringido; un programa en modo usuario no puede acceder directamente al hardware ni a la memoria de otros procesos, lo que protege la estabilidad y la seguridad del sistema.

Cuando un programa en modo usuario necesita un servicio que solo el núcleo puede ofrecer (leer un fichero, reservar memoria, enviar datos por red), realiza una llamada al sistema (system call): una petición formal que provoca un cambio controlado a modo núcleo, donde el SO atiende la petición, y un retorno posterior a modo usuario con el resultado.

Tipos de arquitectura del núcleo

  • Monolítico: todos los servicios del SO (gestión de procesos, memoria, sistemas de archivos, drivers…) se ejecutan dentro de un único bloque de código en modo núcleo. Es el diseño más rápido, porque los componentes se comunican mediante llamadas a función directas, sin cambios de modo.
  • Microkernel: el núcleo se reduce al mínimo imprescindible (comunicación entre procesos y planificación básica) y el resto de servicios se ejecutan como procesos independientes en modo usuario. Es más robusto ante fallos (un servicio que falla no tumba el sistema completo) pero más lento, al requerir más comunicación entre procesos.
  • Híbrido: combina ambos enfoques, manteniendo en el núcleo los servicios más críticos por rendimiento y externalizando otros. Es el diseño más habitual en los sistemas operativos comerciales actuales.
Criterio Monolítico Microkernel
Rendimiento Mayor (comunicación directa) Menor (más cambios de contexto)
Aislamiento ante fallos Menor: un fallo puede afectar a todo el núcleo Mayor: un servicio puede fallar de forma aislada
Complejidad de mantenimiento Mayor con el tiempo, al crecer el núcleo Menor: servicios independientes y modulares
⚠️ No confundir
El núcleo de Linux es monolítico, aunque modular: admite cargar y descargar módulos (por ejemplo, drivers) en caliente sin recompilar el núcleo. Esto no lo convierte en un microkernel: los módulos, una vez cargados, se ejecutan igualmente en modo núcleo y con acceso directo al resto del sistema, a diferencia de los servicios de un microkernel real, que se ejecutan en modo usuario. El núcleo de Windows (NT) y el de macOS (XNU) se consideran híbridos.

4. Administración de memoria

Uno de los cometidos más importantes del SO es gestionar la memoria principal (RAM): decidir qué procesos residen en memoria, en qué dirección, y proteger el espacio de memoria de cada proceso frente al de los demás.

Memoria virtual

La memoria virtual es una técnica que permite a cada proceso trabajar con un espacio de direcciones propio, continuo y aparentemente mayor que la memoria física disponible, con independencia de cómo esté distribuida realmente en la RAM o de si parte de ella reside temporalmente en disco. El SO se encarga de traducir esas direcciones «virtuales» a direcciones físicas reales de forma transparente para el proceso.

Paginación y segmentación

  • Paginación: la memoria se divide en bloques de tamaño fijo. En la memoria lógica del proceso se llaman páginas, y en la memoria física, marcos (frames). Una tabla de páginas, mantenida por el SO, traduce cada página a su marco físico correspondiente.
  • Segmentación: la memoria se divide en bloques de tamaño variable, cada uno correspondiente a una unidad lógica del programa (código, datos, pila…), lo que respeta mejor la estructura del programa, pero complica la gestión del espacio libre.

Cuando el tamaño de los bloques no encaja exactamente con lo que necesita un proceso, aparece la fragmentación: interna, cuando sobra espacio dentro de un bloque asignado (típica de la paginación, al usar bloques de tamaño fijo), y externa, cuando queda espacio libre disperso en huecos demasiado pequeños para ser reutilizados (típica de la segmentación, al usar bloques de tamaño variable).

📌 Nivel básico — imprescindible para el examen
Paginación = bloques de tamaño fijo → puede generar fragmentación interna, pero no externa. Segmentación = bloques de tamaño variable, ajustados a la lógica del programa → puede generar fragmentación externa, pero no interna. La mayoría de sistemas operativos actuales combinan ambas ideas (segmentación paginada).

Cuando la memoria física no es suficiente para todos los procesos activos, el SO recurre al swapping o intercambio: mover páginas o procesos completos entre la RAM y una zona reservada del disco (memoria de intercambio o swap), liberando así RAM para lo que se está usando en cada momento a costa de una pérdida de rendimiento, ya que el disco es mucho más lento que la RAM.

5. Procesos y subprocesos

Un proceso es una instancia de un programa en ejecución: el propio código, más los datos que maneja, más el conjunto de recursos que el SO le ha asignado (memoria, archivos abiertos, tiempo de CPU…). El SO representa cada proceso mediante una estructura de control llamada PCB (Process Control Block), donde guarda su estado, el valor de sus registros, su contador de programa y demás información necesaria para gestionarlo.

Estados de un proceso

A lo largo de su vida, un proceso transita por distintos estados, gestionados por el SO:

  • Nuevo: el proceso se está creando.
  • Listo (ready): el proceso está preparado para ejecutarse y espera a que el planificador le asigne la CPU.
  • En ejecución (running): el proceso está usando la CPU en ese instante.
  • Bloqueado o en espera (waiting): el proceso no puede continuar porque espera un evento externo (por ejemplo, el resultado de una operación de E/S).
  • Terminado: el proceso ha finalizado su ejecución y el SO libera los recursos que tenía asignados.

El tránsito típico es: nuevo → listo → en ejecución → (listo, si se le retira la CPU, o bloqueado, si espera E/S) → en ejecución de nuevo → terminado.

Subprocesos o hilos (threads)

Un subproceso o hilo (thread) es la unidad más pequeña de ejecución dentro de un proceso. Todos los hilos de un mismo proceso comparten su espacio de memoria y sus recursos (archivos abiertos, por ejemplo), pero cada hilo mantiene su propia pila de ejecución y su propio contador de programa, lo que le permite avanzar de forma independiente.

La multihebra o multithreading permite que un mismo proceso realice varias tareas de forma concurrente (por ejemplo, un editor de texto que revisa la ortografía mientras el usuario sigue escribiendo) sin necesidad de crear procesos independientes, que serían más costosos de crear y no compartirían memoria entre sí.

Criterio Proceso Hilo (thread)
Espacio de memoria Propio e independiente Compartido con el resto de hilos del proceso
Coste de creación Alto Bajo
Coste de cambio de contexto Alto Bajo (menos información que guardar/restaurar)
Aislamiento ante fallos Alto: un proceso no afecta a otro Bajo: un fallo puede afectar a todo el proceso
⚠️ No confundir
No debe confundirse la multitarea entre procesos (varios programas distintos ejecutándose «a la vez») con la multihebra dentro de un proceso (un mismo programa dividiendo su trabajo en varios hilos). Ambas se apoyan en el mismo mecanismo de planificación del SO, pero operan a niveles distintos: el proceso es la unidad de asignación de recursos; el hilo, la unidad de ejecución sobre la CPU.

6. Planificación y gestión multitarea

El planificador (scheduler) es el componente del SO que decide, en cada instante, qué proceso o hilo de entre los que están en estado «listo» pasa a ocupar la CPU. Gracias a él, un sistema con una única CPU puede dar la sensación de ejecutar varios programas a la vez: en realidad los va alternando a gran velocidad, en operaciones llamadas cambios de contexto (context switch), en las que el SO guarda el estado del proceso saliente y restaura el del proceso entrante.

Multitarea cooperativa frente a expulsiva

  • Multitarea cooperativa: cada proceso cede voluntariamente el control de la CPU cuando lo considera oportuno. Un proceso mal programado que no cede el control puede bloquear todo el sistema. Modelo típico de sistemas antiguos, como Windows 3.x.
  • Multitarea expulsiva o apropiativa (preemptive): el propio SO puede interrumpir un proceso en ejecución para cederle la CPU a otro, sin depender de su colaboración. Es el modelo de todos los sistemas operativos modernos (Windows, Linux, macOS), y ofrece mucha más estabilidad y capacidad de respuesta.

Algoritmos de planificación clásicos

Algoritmo Funcionamiento Ventaja / inconveniente
FCFS (First Come, First Served) Se atiende a los procesos por orden de llegada Sencillo, pero un proceso largo bloquea a los demás (efecto convoy)
SJF (Shortest Job First) Se prioriza el proceso con menor tiempo de ejecución estimado Minimiza el tiempo medio de espera, pero requiere conocer o estimar la duración
Round Robin Cada proceso recibe un turno fijo de CPU (quantum) y vuelve al final de la cola Reparto equitativo; el rendimiento depende de ajustar bien el quantum
Por prioridades Se ejecuta primero el proceso de mayor prioridad Da respuesta rápida a lo importante, pero puede provocar inanición de procesos de baja prioridad
📌 Nivel básico — imprescindible para el examen
El quantum es el intervalo de tiempo máximo que Round Robin concede a cada proceso antes de forzar un cambio de contexto. Un quantum demasiado corto multiplica los cambios de contexto (y su coste); uno demasiado largo hace que el sistema se comporte casi como FCFS, perdiendo capacidad de respuesta.

7. Entrada/salida

El SO también actúa de intermediario entre los procesos y los periféricos (Tema 8), a través de los controladores de dispositivo (drivers): programas específicos para cada dispositivo que traducen las operaciones genéricas del SO a las órdenes concretas que entiende ese hardware.

Para compensar la diferencia de velocidad entre la CPU y los dispositivos (mucho más lentos), el SO utiliza buffers: zonas de memoria intermedia donde se acumulan los datos mientras se transfieren entre el dispositivo y el proceso.

Métodos de entrada/salida

Método Funcionamiento Ocupación de la CPU
E/S programada (polling) La CPU pregunta activamente y de forma repetida si el dispositivo ha terminado Alta: la CPU queda bloqueada esperando
E/S por interrupciones El dispositivo avisa a la CPU (mediante una interrupción) cuando termina la operación Media: la CPU puede hacer otras tareas mientras espera
DMA (Direct Memory Access) El dispositivo transfiere los datos directamente a memoria, sin pasar por la CPU Mínima: la CPU solo interviene al inicio y al final de la transferencia
📌 Nivel básico — imprescindible para el examen
El DMA (acceso directo a memoria) es el mecanismo más eficiente porque libera a la CPU de la propia transferencia de datos, que se gestiona mediante un controlador DMA dedicado: la CPU solo configura la operación al principio y recibe una interrupción al finalizar. Es un concepto que aparece con frecuencia en preguntas de examen sobre rendimiento de E/S.

8. Sistemas de archivo

El sistema de archivos (file system) es el componente del SO responsable de organizar la información persistente sobre un dispositivo de almacenamiento: cómo se nombran, almacenan, localizan y protegen los ficheros y directorios.

Habitualmente organiza los ficheros en una estructura jerárquica en árbol, a partir de un directorio raíz, y mantiene para cada fichero unos metadatos: tamaño, fechas de creación/modificación, propietario y permisos de acceso.

🔍 Para ampliar — contexto útil, menor probabilidad de pregunta directa
Cada sistema operativo implementa sus propios sistemas de archivo: NTFS y, más recientemente, ReFS en Windows; ext4 en muchas distribuciones Linux; APFS en macOS. Cada uno se estudia con más detalle en el tema del sistema operativo correspondiente (Windows de usuario en el Tema 11, Linux de servidor en el Tema 13). De igual forma, el modelo de permisos de tipo Unix (lectura/escritura/ejecución para propietario, grupo y resto) y las listas de control de acceso (ACL) de Windows se tratan en esos temas.
⚠️ No confundir
Este punto no debe confundirse con el Tema 4 (Ficheros): allí se estudió la organización lógica de la información dentro de un fichero (registros, claves, métodos de inserción y acceso) desde el punto de vista de los datos; aquí se estudia el papel del sistema operativo como gestor del conjunto de ficheros y directorios sobre el almacenamiento.

9. Resumen esquemático

  • Sistema operativo: intermediario entre hardware y programas; gestor de recursos y máquina virtual extendida.
  • Clasificación: monousuario/multiusuario, monotarea/multitarea, uniprocesador/multiprocesador, por lotes/tiempo compartido/tiempo real, y por ámbito (escritorio, servidor, embebido, móvil).
  • Arquitectura: núcleo en modo privilegiado, resto en modo usuario, comunicados mediante llamadas al sistema; núcleos monolíticos (rápidos), microkernel (robustos) e híbridos (combinación habitual hoy).
  • Memoria: memoria virtual como abstracción; paginación (bloques fijos, fragmentación interna) frente a segmentación (bloques variables, fragmentación externa); swapping cuando falta RAM.
  • Procesos y subprocesos: el proceso como unidad de recursos (PCB, estados: nuevo, listo, ejecución, bloqueado, terminado); el hilo como unidad de ejecución, más ligero y compartiendo memoria dentro del proceso.
  • Planificación: cooperativa frente a expulsiva (estándar actual); algoritmos FCFS, SJF, Round Robin (con quantum) y por prioridades.
  • Entrada/salida: drivers y buffers; polling, interrupciones y DMA (el más eficiente, libera a la CPU).
  • Sistemas de archivo: organización jerárquica de ficheros y metadatos; el detalle de cada sistema de archivos concreto se estudia en el tema del SO correspondiente.
